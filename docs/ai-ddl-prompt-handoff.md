# AI DDL Prompt Handoff

## Goal

Improve AI-assisted type mapping in `dmt-rs` so prompt quality is strong across
multiple source/target database pairs without moving to model-authored full DDL.

The desired direction is:

- keep `dmt-rs`'s current architecture where AI returns a type string only
- keep writer-owned DDL generation deterministic
- make the core AI prompt generic
- let database-specific readers/writers/dialects augment that generic prompt

This document is for a follow-on AI or engineer to implement the prompt changes.

## Why This Work Exists

Current `dmt-rs` behavior:

- AI is used only for fallback type mapping
- target writers still build `CREATE TABLE` statements in code
- this is safer than the Go `dmt` repository, which lets AI generate raw DDL

That part is good and should remain.

However, we found a quality problem during E2E validation:

- PostgreSQL source had a user-defined enum type `contact_choice`
- MSSQL target was using AI-assisted fallback mapping
- with a mock provider, the mapping returned `nvarchar(255)` and the DDL was good
- with Anthropic, the mapping returned `varchar(255)`
- the plumbing bug that previously blocked AI output from reaching DDL has now been fixed separately

So the remaining issue is not execution path correctness. It is prompt quality.

## Design Principles

### Keep

- AI returns only a target type string
- SQL / DDL structure is generated by code
- output is validated and sanitized
- database-specific hard invariants should still be enforced in code when confidence is high

### Do Not Do

- do not copy `dmt`'s full model-authored `CREATE TABLE` strategy
- do not expand AI to generate indexes, foreign keys, checks, or raw DDL
- do not make the base prompt MSSQL-specific
- do not embed a giant matrix of vendor rules directly in one global prompt

## Strategy to Borrow From `dmt`

The part worth borrowing from the Go `dmt` repo is prompt augmentation:

- generic core prompt owned centrally
- source-specific context owned by the source side
- target-specific constraints owned by the target side
- optional pair-specific guidance added only when relevant

That is the right idea.

The part **not** worth borrowing is raw AI DDL generation.

## Proposed Architecture

### 1. Generic Core Prompt

Keep a small shared prompt in `crates/dmt-rs/src/ai/prompt.rs` that says, in
effect:

- map the source type to the safest equivalent target type
- preserve semantics over name similarity
- preserve Unicode and character semantics where possible
- prefer lossless mappings
- return only the target type string

This core prompt should remain generic across engines.

### 2. Prompt Augmentation Hooks

Introduce source and target prompt augmentation hooks so database-specific rules
live with the database implementation, not inside the AI module.

Possible shapes:

```rust
pub trait AiPromptAugmentation {
    fn ai_type_mapping_guidance(&self) -> Option<String>;
}
```

Or a more structured context object:

```rust
pub struct AiTypeMappingPromptContext {
    pub source_guidance: Vec<String>,
    pub target_guidance: Vec<String>,
    pub pair_guidance: Vec<String>,
}
```

The specific API can vary, but the ownership model should be:

- source guidance comes from source dialect/reader metadata
- target guidance comes from target dialect/writer metadata
- pair guidance comes from the source→target mapper layer or catalog

### 3. Composition Model

Final prompt should be composed from:

- generic system prompt
- generic user prompt with source type and numeric metadata
- source guidance block
- target guidance block
- optional pair-specific guidance block

This should be assembled in one place so behavior stays predictable.

### 4. Hard Guardrails in Code

Prompting should guide the model, but a few high-confidence invariants should
still be enforced in code after the model returns a type string.

Example category:

- if target is MSSQL and the source type is clearly textual / Unicode-capable,
  normalize or reject `varchar(...)` in favor of `nvarchar(...)`

This is optional for the first implementation, but strongly recommended for
cases where semantic correctness is obvious.

## Initial Concrete Augmentation To Implement

Start with MSSQL target guidance only.

The minimal first target-specific rule should be:

- for textual data targeting MSSQL, prefer Unicode-safe target types
- prefer `nvarchar` / `nchar` over `varchar` / `char` unless non-Unicode intent
  is explicit

This should be written as a semantic rule, not a one-off fix for PostgreSQL
enums.

Good phrasing:

- "When the target database is MSSQL / SQL Server and the source type is textual
  or text-like, prefer Unicode-capable target types (`nvarchar`, `nchar`) over
  non-Unicode types (`varchar`, `char`) unless the source is explicitly
  non-Unicode."

Avoid phrasing like:

- "For PostgreSQL enums always use `nvarchar(255)`"

That is too narrow and not reusable across engines.

## Candidate Source-Side Guidance

For PostgreSQL source, useful prompt augmentation could include:

- user-defined types may be enums or domains
- enums and domains often preserve textual semantics
- unless binary semantics are explicit, unknown user-defined text-like types
  should be treated as text-like, not binary-like

This should remain generic enough to help multiple target engines, not only
MSSQL.

## Files Likely To Change

Minimum likely touch points:

- `crates/dmt-rs/src/ai/prompt.rs`
- `crates/dmt-rs/src/ai/provider.rs` only if prompt arguments need to expand
- `crates/dmt-rs/src/core/catalog.rs` if prompt augmentation is sourced there
- one or more dialect / writer / reader files for augmentation hooks

Potential target-specific augmentation locations:

- MSSQL writer or MSSQL dialect metadata
- PostgreSQL reader or PostgreSQL dialect metadata

## Constraints

- keep changes scoped to prompt construction and prompt metadata flow
- do not rewrite the AI cache design unless needed
- do not change DDL ownership away from writers
- do not regress existing non-AI paths
- do not require every database implementation to provide augmentation on day one

Default behavior should remain valid if no augmentation is present.

## Validation Plan

### Unit / Local Validation

At minimum:

- existing AI tests continue to pass
- add prompt-builder tests for augmentation composition
- add at least one test proving target-specific guidance is injected only when
  relevant

### Focused E2E Validation

Reuse the already-proven fixture pattern:

- PostgreSQL source with enum type `contact_choice`
- MSSQL target in Docker
- AI-enabled migration

Success criteria:

- AI warm-up occurs
- target DDL path uses the AI-derived type
- Anthropic or another live provider should prefer `nvarchar(255)` or another
  Unicode-safe MSSQL text type rather than `varchar(255)`

### Regression Checks

Ensure the prompt remains generic enough that:

- PostgreSQL → MSSQL still works
- MySQL → MSSQL is not made worse by PostgreSQL-specific assumptions
- PostgreSQL → PostgreSQL and MSSQL → PostgreSQL do not get accidental
  MSSQL-specific instructions

## Non-Goals

- solving all cross-database type quality issues in one patch
- introducing full table-level AI DDL generation
- introducing AI-generated finalization DDL
- replacing deterministic type mappers entirely

## Recommended Implementation Order

1. Refactor prompt building to accept augmentation sections.
2. Add no-op default behavior when no augmentation exists.
3. Implement MSSQL target guidance.
4. Optionally add PostgreSQL source guidance for user-defined textual types.
5. Add focused unit tests for prompt composition.
6. Run live-provider E2E validation against the enum fixture.
7. If the model still returns `varchar(...)`, add a small deterministic
   post-processing guard for obvious Unicode-safe MSSQL text cases.

## Summary

The intended end state is:

- generic AI type-mapping prompt
- database-specific augmentation supplied by source/target implementations
- deterministic writer-generated DDL remains unchanged
- better cross-engine prompt quality without adopting `dmt`'s riskier
  model-authored DDL approach
